k8s-wish-system — Architecture

A single LLM call translates natural language into K8s manifests

User
natural language
→
Wish CRD
kubectl wish create
→
wish-grantor
1 LLM call → plan
→
Human review
dry-run by default
→
wish-fulfiller
applies YAML via API
→
Running pods

Safety model

  • Dry-run by default
  • Creator impersonation (SelfSubjectReview)
  • Immutable fulfillment
  • Separate RBAC (grantor vs fulfiller)
  • Shell commands blocked

Any OpenAI-compatible LLM

  • Ollama (local)
  • llama.cpp (local/remote)
  • OpenAI, Azure, etc.
  • One call, structured JSON response

The single-shot limitation

"The LLM gets one chance to be right.
No feedback loop. No cluster awareness. No self-correction."

What it gets right

  • Simple — one call, one plan
  • Auditable — human reviews everything
  • Safe — dry-run default + RBAC
  • Fast — one API call

What it gets wrong

  • No iteration if the plan is wrong
  • No awareness of existing cluster state
  • No feedback from execution results
  • Quality depends entirely on one LLM call

k8s-wish security — what's right

  • Dry-run by default — nothing executes without explicit approval
  • Shell commands blocked — fulfiller rejects CommandType::Shell
  • Creator impersonation via SelfSubjectReview — real K8s identity, not user input
  • Immutable fulfillment — once fulfilled, can't re-execute
  • Separate RBAC — grantor reads, fulfiller writes

But...

Security gaps discovered

#IssueFixResidual risk
1Creator spoofing via kubectl applyMutating admission webhookMust be deployed
2Unrestricted impersonation RBACSubjectAccessReview insteadAudit trail changes
3LLM prompt injectionOutput validation + allowlistsCan't fully prevent
4Permissions ConfigMap not enforcedCode fix in fulfillerResource-level only
5Per-wish LLM endpoint overrideRemove or gate the fieldBreaking change
6Status/plan injectionCEL transition rules + RBACBroad cluster roles
7TOCTOU during fulfillmentCEL transition rulesNone (server-enforced)

CEL transition rules — the fix

Declared in the CRD. Enforced by the API server. No webhook needed.

Spec rules

- rule: "self.wish == oldSelf.wish" message: "wish text cannot be changed" - rule: "!has(oldSelf.creator) || self.creator == oldSelf.creator" message: "creator identity immutable" - rule: "oldSelf.dryRun || self.dryRun == oldSelf.dryRun" message: "dryRun: true->false only"

Status rules

- rule: "!has(oldSelf.plan) || self.plan == oldSelf.plan" message: "plan is immutable once set" - rule: "!oldSelf.fulfilled || self.fulfilled" message: "can't un-fulfill" - rule: "oldSelf.phase != 'Fulfilled' || self.phase == 'Fulfilled'" message: "no phase rollback"

Closes the TOCTOU gap. Can't swap YAML between review and execution.

MCP vs. CLI — do we even need this?

Against MCP for K8s

  • kubectl already works — every LLM knows it
  • Shell is the universal MCP
  • Extra abstraction = extra bugs
  • Documentation is the original MCP

For MCP for K8s

  • Structured output vs. text parsing
  • Safety boundaries — expose only what you want
  • Discoverability — tool listings
  • Composability across systems
"The real value of MCP isn't replacing CLIs —
it's replacing the trust boundary."

Takeaways

  • Single-shot: simple, auditable, needs human review
  • Agentic: powerful, self-correcting, needs trust
  • K8s already has the security primitives (CEL, RBAC, webhooks)
  • MCP complements CLIs — it doesn't replace them
  • The best tool depends on your use case
"The YAML isn't the hard part — the trust boundary is."
